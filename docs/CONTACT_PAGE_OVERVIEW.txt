This document describes the Contact page: purpose, structure, components,
form fields, server action, styling, and how it fits into the AS Brokers site.
The page is the primary lead-capture and consultation request entry point.

--------------------------------------------------------------------------------
1. PURPOSE & USER JOURNEY
--------------------------------------------------------------------------------

The Contact page has three jobs:

  • Capture qualified leads via a structured enquiry form (full name, phone,
    email, topics of interest, consent). Submissions are validated and synced
    to HubSpot with cumulative lead scoring.
  • Set expectations: trust badges (FSP 17273, 25+ years, zero advice fees)
    styled as Apple-esque secure validation tags; "Client Qualification
    Criteria" copy; and "The Actuarial Review Process" three-step section so
    visitors see a structured, institutional path from assessment to allocation.
  • Offer an alternative path: prominent WhatsApp CTA (067 242 9946) and a
    link to "Explore solutions" for users who prefer to browse first.

There is no dropdown in the nav; "Contact Us" in the header links directly to
this page. The page is the main destination for "Book a consultation", "Get
in touch", and "Request a quote" CTAs from Solutions, Calculators, and the
home page.

6. CONTACT ENQUIRY FORM (COMPONENT)
--------------------------------------------------------------------------------

  • Component:  components/forms/ContactEnquiryForm.tsx (client).
  • Server Action: app/(content)/actions/contact.ts — submitContactEnquiry.
  • State:      useActionState(submitContactEnquiry, initialState). Form
                submits via action; state holds success, message, fieldErrors.

  Form container:
  • Card:       bg-[#151518], rounded-[2rem] (Squircle), p-8 md:p-10,
                border border-blue-500/20, shadow-[0_0_40px_rgba(59,130,246,0.12)].
  • Layout:     space-y-6. Hidden input for topics (JSON array). Honeypot
                field "website" (absolute -left-[9999px]) for bot protection.

  Fields (all required unless noted):
  • fullName:   Text input. Label "Full name *". placeholder "Full name".
                Validation: min 2 chars (Zod).
  • phone:      Tel input. Label "Phone *". placeholder "Phone number".
                Validation: min 9 chars.
  • email:      Email input. Label "Email *". placeholder "Email address".
                Validation: valid email.
  • topics:     Multi-select (custom dropdown with checkboxes). Label "What
                would you like to discuss? *". Helper: "Select all that apply.
                We'll prepare for a relevant conversation." Stored as
                JSON.stringify(selectedTopics) in hidden input. Validation:
                at least one topic (array min 1).
  • consent:    Checkbox. name="consent" value="true". Label text: consent to
                transactional messages (appointment reminders, confirmations,
                account notifications); message & data rates; HELP/STOP. Zod:
                literal true required.
  • website:    Honeypot; must be empty (max 0 in schema). If non-empty, action
                returns success without saving (anti-spam).

  Topic options (serviceOptions): Investment Review / Everest Wealth Quote,
  Company Medical & Gap Review, Medical Aid, Gap Cover, Short-term Business
  Insurance, Short-term Personal Insurance, Life Insurance (Business), Life
  Insurance (Personal), Last Will & Testament, Trust & Business Structure,
  Wellness & Integration, Estate Reduction Plan, General enquiry.

  Validation (lib/validations/schema.ts – contactFormSchema):
  • fullName:   z.string().min(2)
  • phone:      z.string().min(9)
  • email:      z.string().email()
  • topics:     z.array(z.string()).min(1)
  • consent:    z.literal(true)
  • website:    z.string().max(0).optional()

  Error display: fieldErrors from action state; amber-400 text below each
  field; aria-invalid and aria-describedby for accessibility. Generic message
  (e.g. "Please fix the errors below.") and action-level message shown when
  !state.success.

  Submit button: "Initiate Wealth Engineering Request". Full width, white bg, black text,
  font-bold, py-4, rounded-xl, hover:bg-zinc-200. Disabled when isPending;
  label changes to "Sending…".

  Success state: When state.success is true, form is replaced by a success
  card (same rim/card style). Green check icon, "Request received" heading,
  short message (personal response, phone or WhatsApp, not a call centre).
  CTA: "WhatsApp us in the meantime" (wa.me link, green button).

--------------------------------------------------------------------------------
7. SERVER ACTION & INTEGRATIONS
--------------------------------------------------------------------------------

  • Action:     submitContactEnquiry(prevState, formData). useActionState
                passes previous state and form data.
  • Steps:      (1) Parse formData (fullName, phone, email, topics JSON,
                consent, website). (2) contactFormSchema.safeParse. (3) If
                validation fails, return flatten() fieldErrors and message.
                (4) If website non-empty (honeypot), return success without
                persisting. (5) syncContactToHubSpot(payload) – creates/updates
                contact in HubSpot, cumulative lead scoring. (6) If HubSpot
                fails, return error message. (7) Optionally trigger Trigger.dev
                task "generate-financial-pdf" with email and fullName (if
                TRIGGER_SECRET_KEY set). (8) Return success and "Thank you.
                We'll be in touch."
  • HubSpot:    lib/hubspot.service – syncContactToHubSpot. Used for CRM and
                lead scoring.
  • Trigger.dev: Optional one-off trigger for PDF generation; failure only
                 logged in development.

10. DATA & CONTENT
--------------------------------------------------------------------------------

  Advice Fees on Investments.

11. RESPONSIVE BEHAVIOUR
--------------------------------------------------------------------------------

  • Main section:  Single column on small; two columns from lg (grid-cols-2).
                   Left column stacks headline, copy, image, spacer, then "We
                   work best with" at bottom. Right column form full width
                   below on small.

  • What happens next: One column on small; three columns from sm (grid-cols-3).

  • Form:          Full width within column; no horizontal scroll. Topics
                   dropdown max-h-56 overflow-y-auto on small screens.

--------------------------------------------------------------------------------
12. ACCESSIBILITY & COMPLIANCE
--------------------------------------------------------------------------------

  • Labels:       All inputs have associated <label> with htmlFor/id. Consent
                  uses wrapping label.
  • Errors:       aria-invalid and aria-describedby when fieldErrors present;
                  id on error paragraph (e.g. fullName-error). Generic
                  message has role="alert".
  • Honeypot:     website field aria-hidden, tabIndex=-1, off-screen so
                  screen readers and users don't interact.
  • Consent:      Copy mentions HELP/STOP for opt-out (common for SMS/marketing
                  compliance). Form is for enquiry and transactional follow-up.
  • Links:        External WhatsApp: target="_blank" rel="noopener noreferrer".
